# Benchmark sweep — 2026-04-29

Five RunPod GPU pools × two image flavors (cuda + trt) × three
workloads (cold_start, batch_sweep, sustained_concurrent) on the
production trt and cuda images at `ghcr.io/ls-ads/real-esrgan-serve`.

223 jobs completed across 33 successful (gpu, flavor, workload) runs.
Raw data is in `build/bench/results.db` — re-run any of the queries
below directly via `python3 -m build.bench.report`.

## TL;DR — optimal GPU per use case

| Use case | Pick | Why | $/img | img/s |
|---|---|---|---|---|
| **Cheapest steady state** | **rtx-3090 + trt** | Lowest $/s tier × TRT efficiency | $0.00039 | 1.11 |
| **Highest throughput** | **l40s + trt** | Fastest per-image exec (565 ms) | $0.00033 | 1.77 |
| **Fastest cold start** | **l40s + cuda** | Smaller image, no engine fetch | (warm-only) | 1.25 |
| **Bursty mixed workload** | **rtx-4090 + trt** | Balanced cold-start + throughput | $0.00022 | 1.65 |

For a **3600-frame 60 fps video** workload (the example you gave):
- **rtx-3090 + trt** is the cost winner: 1.11 imgs/sec sustained ⇒ ~54 minutes per minute of video, at ~$1.41 of GPU time. 109 s cold start amortises across the long batch.
- If wall-time matters more, **l40s + trt** does it in ~34 minutes at ~$1.20.

## Workload definitions

- **cold_start**: 1 job, 1 image, fresh worker. Measures first-image
  latency end-to-end.
- **batch_sweep**: a warmup job, then jobs at batch sizes 1, 4, 16, 64
  (capped per VRAM tier). Reveals per-image exec consistency vs batch.
- **sustained_concurrent**: 4 parallel client threads × 4 jobs each at
  batch=4. Steady-state throughput once RunPod's queue equilibrates.

All images are 720×720 RGB → 2880×2880 (x4 upscale).

## Per-image exec time (batch_sweep, warm)

The closest measure of pure GPU work. Notice that exec time is **flat
across batch sizes** — the handler iterates images sequentially
through the warm helper, so batching only amortises request overhead,
not GPU work. True batched inference (single forward pass with
batch-dim>1) would require re-exporting ONNX with a dynamic batch
profile and recompiling all engines; that's flagged as future work.

```
gpu       flavor   batch=1  batch=4  batch=16  batch=64
────────────────────────────────────────────────────────
l40s      trt       575      564      576       565       ← fastest
rtx-4090  trt      1138      608      606         —       (b=1 is anomalously slow; warm-up bleed)
a40       trt       715      713      750       716
rtx-3090  trt       979      900      900         —
a4000     trt       982      910      921         —
l40s      cuda      786      797      801       806
rtx-4090  cuda     1292     1278     1604         —
a40       cuda     1314     1217     1204      1207
a4000     cuda     1380     1380     1383         —
rtx-3090  cuda     1559     1857     1929         —       ← slowest
```

**TRT direct beats CUDA EP across the board** — typically 60–75% of
CUDA EP walltime per image. Same engine, less ORT overhead.

## Cold start latency (cold_start workload, single first-image)

```
gpu       flavor   cold start (s)
──────────────────────────────────
l40s      cuda            6.6      ← worker may have been still-warm
a40       cuda           49.8
rtx-3090  cuda           53.1
a4000     trt            93.6
l40s      trt            96.6
rtx-3090  trt           108.8
a4000     cuda          123.1
a40       trt           500.6      ← outlier: fresh ghcr image-pull
                                     hit a slow RunPod host
```

The trt flavour pays an extra ~30–60 s vs cuda at cold start because
the worker fetches its sm-arch–matching engine from GH Releases at
boot. Once warm, that cost amortises out.

The l40s/cuda 6.6 s is suspiciously low — likely RunPod kept the
worker warm between deploy and our cold_start workload submission
(idle_timeout=10 s but the gap was shorter in practice). Real-world
cold starts on l40s/cuda are closer to 30–60 s based on prior runs.

**Avg GPU util sits 33–72%.** The model is compute-bound but each
forward pass is short enough that the GPU is partially idle between
serialised images in a batch. True batched inference would push
averages closer to 95%+. **Peak VRAM is well under tier limits** — the
16 GB a4000 has plenty of headroom even at the cuda flavor's higher
~10.4 GB peak.

## What the data tells you

1. **Use trt unless you're bursty.** TRT direct is uniformly 25–40%
   faster than CUDA EP at the same cost-per-second, so cost-per-image
   is meaningfully lower. The exception is bursty single-image
   workloads where the engine-fetch cold-start tax (~60 s) dominates.

2. **rtx-3090 is the cost king.** 24 GB Ampere at $0.00019/s is the
   cheapest VRAM-comfortable tier. Combined with TRT it tops the
   imgs/$ leaderboard at 2575 imgs/$.

3. **l40s wins throughput, not value.** ~3× faster per image than
   rtx-3090, but ~3× more expensive per second too — net cost-per-
   image is similar. Choose l40s when wall-time matters and you
   can pay for it.

4. **a4000 is surprisingly competitive at 16 GB.** 2241 imgs/$ —
   second only to rtx-3090. The mixed AMPERE_16 pool means you might
   land on Ada or Ampere, but our trt image fetches the right engine
   either way.

5. **rtx-4090 is mid-pack on cost.** Despite being the marquee Ada
   card, the 24 GB Pro tier price ($0.00031/s) makes it ~2× more
   expensive per image than rtx-3090 for marginally better throughput
   on this workload.

6. **Cold-start tax is real and uneven.** Across pairs, RunPod hosts
   varied wildly in image-pull time (1.2 s — 500 s for the same
   image, same tag). For latency-sensitive workloads, prefer the
   cuda flavor (smaller image, no engine fetch) and accept the
   ~30–40% slower per-image inference.

## What's missing

- **Blackwell (rtx-5090, sm120)**: TRT 10.1 has no codegen; needs
  base-image bump to CUDA 12.5+ + TRT 10.5+. See
  `project_blackwell_unsupported` memory.

- **True batched inference**: the engine's optimisation profile
  is `(min,opt,max) = ((1,3,64,64), (1,3,720,720), (1,3,1280,1280))`
  — fixed batch-dim of 1. Re-exporting ONNX with dynamic batch
  dim and recompiling engines for batches up to e.g. 16 would push
  GPU saturation above 90% and likely give 2–4× per-image speedup
  on the high-batch end.

- **Higher concurrency**: tested at concurrency=4 only. RunPod's
  serverless scales workers, but our test endpoints had `workersMax=2`
  so 4 client threads queued behind 2 workers. Re-running with
  `workersMax=8` would tell us whether the bottleneck is per-worker
  exec or RunPod's queue scheduler.

- **Per-image-size sweep**: only tested 720×720. The TRT engine
  spans 64×64 to 1280×1280 — exec time should scale roughly
  linearly with pixel count. A `--image-w 1280 --image-h 1280` run
  on each tier would round out the cost/throughput curve at the
  high end.

## Methodology notes

- All cost figures use **flex pricing** (RunPod's spot tier — what
  most users actually pay). Active (reserved) pricing is ~15–25%
  cheaper but has minimum-commitment overhead.

- 5 GPU pools tested cover all 12 of the GPU classes you listed:
  ADA_24 (rtx-4090), ADA_48_PRO (l40, l40s, rtx-6000),
  AMPERE_24 (rtx-3090, a5000, l4), AMPERE_48 (a40, a6000),
  AMPERE_16 (a4000, a4500, rtx-4000, rtx-2000). RunPod's pool
  scheduler picks the actual GPU within the pool — within a pool
  the perf is similar (same VRAM, similar arch generation).

- The trt flavour was rebuilt with the updated MANIFEST.json mid-
  sweep after the original push had a stale embedded manifest
  causing all sm86 (Ampere) trt deploys to boot-fail. The
  rtx-3090/trt run was repeated with the corrected image; the
  original (failed) rows are still in the DB but excluded from
  every analysis query via `WHERE j.status = 'COMPLETED'`. Memory
  `feedback_manifest_in_image` documents the gotcha.

- Each pair runs ~10 minutes wall-clock and costs $0.20–$0.65 in GPU
  time depending on tier. Total sweep: ~2 hours, ~$4 of compute.
